## 7. Strategic

Larger directional gaps or decisions that affect the project roadmap.

| # | Issue |
|---|-------|
| ~~S1~~ | **Won't fix — by design.** DABs does not support Unity Catalog resources natively (catalog, schema, volume, grants); Terraform is mandatory for those regardless. For workspace resources (clusters, warehouses, etc.) that DABs does support, the native `databricks.yml` definition is sufficient — Laktory's value-add (pipeline DSL, DAG, multi-orchestrator) is specific to pipelines. The hybrid strategy (DABs for pipelines + Terraform for UC/account resources) is documented in `docs/concepts/dab.md`. |
| ~~S2~~ | **Won't fix.** Adding a new resource is 2 lines (import + field) and happens rarely enough that a generator or registry pattern adds more complexity than it saves. |
| ~~S3~~ | **Done** — added `register_nw_extensions: bool` to `Settings` (env var `LAKTORY_REGISTER_NW_EXTENSIONS`, default `True`); `narwhals_ext/__init__.py` now skips all three extension registrations when the flag is `False`. |
| ~~S4~~ | **Done** — added `model_validator` to `TableDataSink` (+ `HiveMetastoreDataSink` by inheritance) rejecting Polars at construction time with a clear error pointing to `FileDataSink`; removed unreachable `_read_polars` / `_write_polars` stubs and orphaned `LazyFrame` imports from `UnityCatalogDataSource` and `UnityCatalogDataSink` (construction-time guard from U1 already made those methods dead code). |
| ~~S5~~ | **Won't fix — by design.** `access_controls` (workspace operations) and `grant`/`grants` (Unity Catalog data privileges) are orthogonal at the Databricks API level; merging them would add routing complexity without simplifying the user model. `docs/concepts/governance.md` now has a decision table and worked examples that make the distinction clear. |
